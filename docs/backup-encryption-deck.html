Oqto Backup & Data Encryption Strategy

How much should we encrypt? Options A through D compared.
Reject

A. Encrypt secrets only

Encrypt obvious secret files (auth.json, eavs.env, keys.db, setup-state.env) in backups. Leave history DBs, session JSONL, and workspace code as cleartext.

ProsCons
+ Simplest to implement
+ Fastest restore		– Chat logs / prompts exposed on backup theft
– Workspace code exposed
– Inconsistent mental model
Verdict: Reject — chat logs contain sensitive business logic and personal data. Treating them as "non-secret" is a classification error.
Recommended baseline

B. Encrypt everything in backups

Encrypt every backup archive with a per-user symmetric key. Contents are decrypted only at restore time on the target host.

ProsCons
+ Uniform policy: backed up → encrypted
+ Single key per user (easy rotation)
+ Safe on any untrusted object store		– Key escrow problem
– Granular restore decrypts whole archive
Verdict: Accepted — highest impact, lowest operational cost. Defends the most likely threat (backup theft).
Opt-in hardening

C. Encrypt all data at rest + backups

Use SQLCipher for every SQLite DB. Encrypt config files individually. Backups are encrypted archives of already-encrypted files.

ProsCons
+ Defense in depth (disk + backup)
+ Per-database passphrases possible
+ Compliance friendly		– SQLCipher compile-time dep change
– ~5-15% SQLite perf hit
– Key management explosion
Verdict: Opt-in — enable via feature flag for compliance-sensitive multi-user deployments.
Out of scope

D. End-to-end fortress

Strictest: SQLCipher at-rest + encrypted backups + mTLS everywhere + short-lived keys + HSM.

ProsCons
+ Maximum security posture
+ Audit-friendly		– High implementation cost
– Operational complexity
– Overkill for self-hosted
Verdict: Defer — only justified for multi-tenant SaaS or regulated enterprise. Can add later without breaking B/C.

Proposed Architecture: B + Optional C

├── Oqto Host │ │ Live data (cleartext by default) │ ├─ oqto-log.sqlite │ ├─ hstry.db │ ├─ eavs keys.db / providers.db │ └─ Pi session JSONL files │ │ oqto-backup daemon: │ 1. sqlite3 .backup → /tmp/snapshots/<db>.bak │ 2. tar + age → /tmp/snapshots/<user>.tar.age │ 3. upload to object store │ │ Encrypted backup archive (.tar.age) │ ├─ Envelope: DEK encrypted by user master key │ └─ Inner: snapshot manifest + checksums + files │ │ | │ ▼ │ ├─────────── Object Store (untrusted) ────────────┘

Key Model

Per user: one master backup key (256-bit, age or AES-GCM) stored in systemd-creds or password manager. Per backup: random data-encryption key (DEK) DEK encrypted by master key → envelope in manifest Benefits of envelope encryption: • Rotate master key without re-encrypting all historical backups • Unique DEK per backup → compromise isolation • Platform backups use separate platform master key
